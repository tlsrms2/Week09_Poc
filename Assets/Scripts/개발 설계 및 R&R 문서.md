# 덱빌딩 마법 퍼즐 POC 개발 설계 및 R&R 문서

## 1. Why (개발 목적 및 아키텍처 의도)
* **목적:** '2차원 공간 배치'와 '문양 겹치기를 통한 위력 증폭'이라는 핵심 퍼즐 로직이 실제 전투에서 재미있는 시너지를 내는지 검증한다.
* **아키텍처 의도:** 3명의 개발자가 Git 충돌 없이 병렬로 작업할 수 있도록 핵심 로직(Grid), 턴 제어(Combat), 시각화(UI/UX)의 결합도를 최소화(Decoupling)한다.

---

## 2. How (핵심 시스템 설계 - Unity / C#)
* **데이터 주도 설계 (ScriptableObject):**
  * 스킬의 형태, 문양, 위력 등은 기획 데이터로 분리하여 관리.
  * `BlockData` (형태 및 `SymbolType` 배열) / `CardData` (기본 위력 및 타입)
* **그리드 데이터 구조 (`SymbolType[,]`):**
  * 공간 연산의 최적화를 위해 단순 `bool`이 아닌 열거형 2차원 배열을 사용.
  * `TargetGrid[x, y] == Block[x, y]` 방식으로 직관적인 겹치기 판별.
* **이벤트 기반 통신 (Observer Pattern):**
  * 모듈 간 직접 참조를 피하고, 전역 이벤트를 통해 상태 변화를 알림 (`OnBlockPlaced`, `OnTurnEnded` 등).
* **상태 기계 (State Machine):**
  * 턴 꼬임 방지를 위한 전투 흐름 제어 (`PlayerDraw -> Placement -> Resolution -> EnemyTurn`).

## 3. What (모듈별 역할 분담 - R&R)

총 3인 체제로 각자의 도메인을 철저히 분리하여 담당한다.

### 👑 Lead : Core System & Combat Flow
전체 프로젝트의 기반(Architecture)을 구축하고, 게임의 시작부터 끝까지의 흐름을 제어한다.
* **시스템 기반 구축:** 
  * `ScriptableObject` 구조 설계 및 데이터 에셋 세팅.
  * 시스템 간 통신을 위한 글로벌 이벤트 매니저(Event Bus) 구현.
* **턴 상태 기계(State Machine):** 
  * 플레이어와 적의 턴 루프, 상태 전이 로직 작성.
* **덱 및 전투 매니저:** 
  * 덱 셔플, 드로우, 패(Hand), 무덤 관리 시스템.
  * 턴 종료 시 도출된 최종 데미지를 적과 플레이어의 HP/방어도에 적용하는 결산 처리.

### 👨‍💻 Dev A : Grid & Puzzle Logic (코어 알고리즘)
시각적인 UI 처리는 배제하고, 순수하게 데이터 기반의 수학적/논리적 연산에만 집중한다.
* **Grid Manager 생성:** 
  * 2차원 `SymbolType` 배열 데이터 구조화 및 현재 그리드 상태(배치된 블록들) 추적.
* **배치 및 겹치기 검증 (핵심):** 
  * 마우스 좌표를 그리드 인덱스로 변환하여 블록이 그리드를 벗어나는지 검사.
  * 블록이 기존 블록 위에 겹칠 때, 상하 칸의 **문양(SymbolType)이 100% 일치하는지 판별**하는 알고리즘 개발.
* **위력 증폭 연산자:** 
  * 턴 종료 이벤트 발생 시, 그리드 전체를 스캔.
  * 겹쳐진 칸 수를 계산하여 스킬의 최종 배수(Multiplier)를 연산하고 Combat System에 반환.

### 👨‍💻 Dev B : UI/UX & Interaction
유저의 마우스 입력과 화면에 출력되는 모든 시각적/직관적 피드백을 책임진다.
* **마우스 인터랙션 (Drag & Drop):** 
  * 하단 패(Hand)에 있는 블록을 마우스로 집어 그리드 위에 올리고 스냅(Snap)시키는 조작감 구현.
* **동적 시각 피드백:** 
  * 유저가 블록을 들고 이동할 때, Dev A의 검증 로직을 호출하여 배치 가능 여부를 반투명 색상(초록/빨강)으로 실시간 렌더링.
* **정보 표기 직관화 (TextMesh Pro 활용):** 
  * 겹쳐진 마법 블록에서 현재 몇 배의 증폭이 일어나고 있는지, 최종 데미지가 얼마인지 TextMesh Pro를 이용해 명확하게 화면에 렌더링.
  * 턴 결산 및 데미지 발생 시의 UI 연출.

## 4. Learned (POC 검증 목표 및 회고 계획)
POC 개발 완료 후 다음 항목들을 중점적으로 평가하고 회고한다.
* **재미 검증:** 겹치기 규칙이 유저에게 흥미로운 퍼즐적 압박감을 주는가?
* **직관성 검증:** 블록이 여러 겹 쌓였을 때, 유저가 아래에 깔린 문양과 현재 배수를 쉽게 인지할 수 있는가?
* **확장성 평가:** 현재의 `SymbolType[,]` 구조가 향후 맵 크기 변화나 특수 기믹 타일 추가 시에도 유연하게 대응할 수 있는가?
